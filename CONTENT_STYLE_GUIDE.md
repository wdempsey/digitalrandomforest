# Content Style Guide

This document describes how writing in the digital garden should look and feel.

The goal is not rigid rules, but a consistent voice across essays, notes, patterns, and library entries.

---

# Writing Tone

Writing should aim to be:

- precise
- thoughtful
- reflective
- intellectually honest
- readable to technically literate audiences

The tone should resemble:

- a researcher thinking in public
- a notebook slowly becoming essays
- careful reasoning rather than hype

Avoid:

- exaggerated claims
- startup-style hype language
- aggressive certainty
- unnecessary jargon

Prefer clarity and curiosity.

---

# Writing Structure

Most pages should use a simple structure.

Typical pattern:

Title

Short introductory paragraph explaining the idea.

One or more sections expanding the idea.

Occasional equations, code blocks, or diagrams when helpful.

Closing reflection or open question.

Avoid extremely long unbroken sections of text.

---

# Essays

Essays are longer, more developed pieces.

They may contain:

- arguments
- reflections
- explanations of complex ideas
- conceptual frameworks

Essays may evolve over time.

Essays can include a minimal sketch or illustration.

---

# Notes

Notes are smaller and more exploratory.

Notes may include:

- partial explanations
- questions
- observations
- early-stage ideas

Notes should still be readable but do not need to be fully polished.

---

# Patterns

Patterns describe reusable ideas.

Examples:

- AI workflow design patterns
- research workflow patterns
- thinking tools
- organizational practices

Patterns should emphasize practical insight.

---

# Library

Library pages are the most academic content.

These include:

- literature notes
- paper summaries
- methodological commentary
- academic references

Library entries should be careful and precise.

They should summarize ideas clearly without pretending to be formal publications.

---

# Equations

Mathematics is expected on this site.

Equations should be rendered using LaTeX.

Inline math example:

$E[Y^1 - Y^0]$

Block math example:

$$
ATE = E[Y^1 - Y^0]
$$

Mathematical notation should be clear but not excessive.

Use equations when they genuinely help explanation.

---

# Code

Code may appear in notes or essays.

Code should be:

- concise
- illustrative
- readable

Avoid large code dumps.

Short, focused examples are better.

---

# Linking

Internal links should be used to connect ideas across the garden.

Notes should naturally reference related notes or essays.

Over time this should create a network of ideas.

---

# Versioning

Pages may evolve.

Metadata may include simple version labels such as:

v1  
v2  
vFinal  
vFinal-final

These are meant to be slightly playful but should remain subtle.

---

# Sketches

Essays may include one minimal sketch.

Sketches should feel:

- quiet
- exploratory
- notebook-like

They should complement the writing rather than dominate the page.
